With all of the album IDs, we can work on getting all of the song data we need. To do this, we have to call the api endpoint for audio features on each song of each album, collapsing all of this into one data frame. First, we hit the album/{id}/tracks endpoint to get the song ids for all of the songs on the album we are getting data for. Then we hit the audio-features endpoint giving it all the ids of the songs on the album that we just got. My method for collapsing all of this data is to get the song audio features as a separate data frame each time, then add that data frame to a list, in this case audio_df_list. We will see the merging of all the data frames in the next part. I can’t show you what the intermediate data frames look like since this is all in a loop but feel free to add print statements or try out the api calls with one id to see what the data looks like at each step.

library("tidyverse")

audio_df_list <- c()
i <- 0

for (id in ids) {
  #Getting the track data for each song on the album, we really want the track id for each song
  track_data <- httr::GET(paste('https://api.spotify.com/v1/albums/', id, '/tracks', sep=""), 
      accept_json(), 
      add_headers('Authorization' = key))
  
  track_data <- fromJSON(txt=content(track_data, as="text"))
  
  #Get the track ids as a string that meets the format for requesting multiple ids on Spotify
  track_ids <- paste(track_data$items$id, collapse="%2C")
  #Get the track name to combine with the audio features
  track_names <- track_data$items$name
  
  #Get the audio data for all the songs on the album, this has many attributes which I talk about in the next section
  audio_data <- httr::GET(paste('https://api.spotify.com/v1/audio-features?ids=', track_ids, sep=""), 
      accept_json(), 
      add_headers('Authorization' = key))
  
  
  audio_data <- fromJSON(txt=content(audio_data, as="text"))
  
  #add songnames to the audio data
  audio_data$audio_features$songtitle <- track_names
  
  #Get the track data for the song, this includes 2 pieces of info we need which are explicit and popularity of the song
  song_data <- httr::GET(paste('https://api.spotify.com/v1/tracks?ids=', track_ids, sep=""),
      accept_json(),
      add_headers('Authorization' = key))

  song_data <- fromJSON(txt=content(song_data, as="text"))

  audio_data$audio_features$explicit <- song_data$tracks$explicit
  audio_data$audio_features$popularity <- song_data$tracks$popularity
  
  #add to list
  audio_df_list[i] <- audio_data
  i <- i + 1
  
}

To get our final desired one data frame, we just need to merge everything. Using the reduce function we can merge each data frame from the list all in one call leading to the final data frame. I suggest looking at the Spotify documentation to understand what these metrics mean, found here

all_audio <- Reduce(function(x, y) merge(x,y,all=TRUE), audio_df_list)

Prediction

To predict whether or not a song is explicit, lets make a k nearest neighbors model. Here we will make a model and do a quick test with a random split of training and test data. I have decided to select 5 features to use in the model, you can feel free to experiment with the features used if you want to explore more. For the tutorial, I also selected an arbitrary k value to make the model look at the 5 nearest neighbors to determine the class (explicit or not). If you are interested, it would be better to put a lot more effort into choosing the best k value, since k is actually a hyperparameter and performing parameter tuning to minimize overfitting or underfitting could have a large effect on the results. If you want to explore this idea, the K nearest neighbors algorithim and its applications, I suggest checking out this resource

Since in the Exploratory Data Analysis we saw explicit songs had a very different popularity value compared to non explicit songs, lets explore the difference between a k nearest neighbors model that does include popularity as a feature and a k nearest neighbors model that does not.

First we should select the features we want to use, in addition to songtitle for sampling the dataset to make test and training data.

set.seed(1000)

audio_feats <- all_audio %>% select(energy, danceability, loudness, popularity, songtitle, explicit)

testdata <- audio_feats %>%
  sample_frac(.2) 

traindata <- audio_feats %>%
  anti_join(testdata, by="songtitle")

With the training and testing data built, its time to make to kNN models. We want to use all available features, the test and training data, k=5, and set norm = TRUE so that the data will be normalized for the model. For the first we include popularity and the second we do not.

tk <- kNN(explicit~., traindata %>% select(-songtitle), testdata %>% select(-songtitle), norm=TRUE, k=5)
nopop <- kNN(explicit~., traindata %>% select(-songtitle, -popularity), testdata %>% select(-songtitle, -popularity), norm=TRUE, k=5)

Lets look at the performance table, showing the predictions versus the actual labels (explicit vs not explicit)

These preliminary results show a very successful predictor with popularity included but a not so much successful classifier that excludes it. Lets follow up with a more intense verification procedure of cross validating the 2 models.

Cross Validation

To start we will assume a null hypothesis of there being no significant difference in the performance of the two models. To perform a 5 fold cross validation experiment we need to use the cvTools package to easily select the test and train data for each fold. Then we go ahead and build predictions using each model, calculate the error rate, and combine the error rate for both models into a data frame each fold. (test modeled from here)

library(cvTools)
set.seed(1111)
fold_indices <- cvFolds(n=nrow(audio_feats), K=5)

error_rates <- sapply(1:5, function(fold_index) {
  #get indices for test set for current fold
  test_indices <- which(fold_indices$which == fold_index)
  
  test_set <- na.omit(audio_feats[test_indices,])
  train_set <- na.omit(audio_feats[-test_indices,])

  #build models
  pop_fit <- kNN(explicit~., train_set %>% select(-songtitle), test_set %>% select(-songtitle), norm=TRUE, k=5)
  nopop_fit <- kNN(explicit~., train_set %>% select(-songtitle, -popularity), test_set %>% select(-songtitle, -popularity), norm=TRUE, k=5)
  
  #calculate error
  pop_error <- mean(test_set$explicit != pop_fit)
  nopop_error <- mean(test_set$explicit != nopop_fit)
  
  
  c(pop_error, nopop_error)
  })
rownames(error_rates) <- c("Popularity", "No Popularity")
error_rates <- as.data.frame(t(error_rates))

Lets look at a plot of the error rates for each fold for the 2 different models

error_rates <- error_rates %>%
  mutate(fold=1:n()) %>%
  gather(method,error,-fold)


dotplot(error~method, data=error_rates, ylab="Mean Prediction Error")

The last step for us is to check whether there is a significant difference in the two models. If our p value for the relationship between error and the method used is less than an alpha level of .05, we can reject the null hypothesis that the two models are not different (more on p testing here)

lm(error~method, data=error_rates) %>% 
  broom::tidy() %>%
  knitr::kable()
term estimate std.error statistic p.value

The p value is very small, which is far under the alpha level. This means we can reject the null hypothesis that there is no difference between the two models. This makes sense as the popularity one performed really well in comparison to the one without popularity.
